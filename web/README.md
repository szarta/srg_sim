# srg-supershow web — M-R2 presentation layer

A **purely presentational** browser client over `srg-core`, compiled to WASM. The same
Rust crate that runs natively server-side (authoritative PvP) drives the match here; the
UI only renders [`observable_state`](../src/state.rs) and submits decisions back through
the [`Session`](../src/session.rs) — no game logic lives in JavaScript.

Stack matches the card-search frontend (`~/data/srg_card_search_website/frontend`):
**Vite + React 19 + Tailwind v4**.

## Build & run

```bash
# 1. Build the WASM package (from the repo root) — writes web/src/pkg/
invoke wasm
#    (one-time setup: rustup target add wasm32-unknown-unknown
#     and cargo install wasm-bindgen-cli --version <wasm-bindgen crate version>)

# 2. Install deps and start the dev server
cd web
npm install
npm run dev
```

`src/pkg/` (the wasm-bindgen output) is generated by `invoke wasm` and git-ignored —
rebuild it whenever `srg-core` changes.

## What it shows (this cut)

- Board-vs-board: the opponent's mat and yours (competitor, in-play, deck size, hand /
  hidden hand count), rendered from `observable_state`.
- An **inspectable discard stack** per side (click to fan the pile).
- The outstanding decision as clickable options; you play seat **A**, seat **B** is a
  local heuristic AI, so every surfaced step is your move (or the result).
- A seed box + "New match" to replay deterministically.

Decks are bundled sample enriched-`Deck` JSON in `src/sample/` (extracted from a
conformance fixture). In production the client syncs the card DB as JSON from
get-diced.com; wiring that fetch, and the **drag-drop** card interactions (play by
dragging onto the mat), are the next increment on this foundation.

## Data flow

`WasmSession.open(deckA, deckB, seats, seed)` → `.step()` / `.submit(i)` return the
same `{kind, request|result}` JSON the `srg session` CLI emits (one wire contract,
`Step::to_json`). `.snapshot()` / `WasmSession.restore()` make a match a pure function
of its snapshot — resumable and reproducible.
